# Palletizer

Исследовательский репозиторий **трёхмерной паллетизации**: последовательная укладка коробок на одну или несколько паллет с учётом поворотов, ограничений по устойчивости (опора, центр масс) и набора метрик качества. Код на **C++20**, сборка через **CMake**; набор из **436** CSV-задач в `tests/`, мультипаллетные сценарии в `multitests/`. Текст выпускной работы, библиография и рисунки — в каталоге `papers/` (в т.ч. реферат с формулировкой целей и итогов экспериментов).

## Возможности

- Солверы: жадная укладка, **DBLF**, **FFD**-препроцессинг, **LNS** (поиск в большой окрестности на кодировке порядок + поворот + привязка к углу), **генетический** (популяция + мутации; см. таблицу в **[ALGORITHMS.md](ALGORITHMS.md)** и разделы `papers/4-1-EYU.tex`, `papers/3-Obzor.tex`).
- **Мультипаллета**: в данных задаётся `pallet_count`, ответ хранит несколько списков размещений; в CSV ответа первая колонка — `Pallet` (индекс с нуля).
- **Склейка сцен** для стресс-тестов: `TestData operator+`, загрузка каталога `load_multitest_combined("multitests/<id>")`, генератор `bin/generate_multitests`.
- **Бенчмарк** в `src/main.cpp`: параллельный прогон всех `tests/*.csv` и всех подкаталогов `multitests/*/`, сводка с **90% доверительным интервалом**, выгрузка `answers/metrics.csv` и `answers/multitests_metrics.csv`, строка `BENCHMARK_CSV_LINE` / `MULTITESTS_BENCHMARK_CSV_LINE` для агрегации прогонов.

## Сборка и запуск

Требования: **CMake ≥ 3.16**, компилятор с **C++20**, потоки POSIX (`Threads`).

Из корня репозитория (чтобы находились каталоги `tests/`, `multitests/`):

```bash
cmake -S . -B build -DTHREADS_NUM=96
cmake --build build
./bin/main
```

Опции CMake:

- `THREADS_NUM` — число рабочих потоков в бенчмарке (по умолчанию в `CMakeLists.txt` задано **96**; при необходимости уменьшите под свою машину).
- `ENABLE_ASSERT` — ассерты в коде (`ON`/`OFF`).

Собираемые цели: **`main`** (бенчмарк), **`test_height_handler`**, **`train_lns_solver_mutations`**, **`generate_multitests`**.

По умолчанию в `main()` вызывается **`LNSSolver`** для одиночных и мульти-тестов; лимит времени на один запуск солвера задаётся константой **`TIMELIMIT`** в `src/main.cpp` (сейчас **10 000** мс на тест). Чтобы прогнать другой солвер, замените шаблонный аргумент в вызовах `launch_solvers<…>` / `launch_multitests_benchmark<…>` на нужный класс из `src/solvers/`.

## Структура данных (кратко)

| Сущность | Назначение |
|----------|------------|
| `Box` | SKU, количество, габариты, вес, strength, aisle, caustic |
| `TestDataHeader` | Размеры паллеты, допустимые повороты, веса в целевой функции скоринга LNS/GA (`score_*`, `score_normalization_height`) |
| `TestData` | Заголовок + список коробок + **`pallet_count`** |
| `Position` | Размещение: SKU и две противоположные вершины параллелепипеда `(x,y,z)` … `(X,Y,Z)` |
| `Answer` | **`std::vector` паллет**, на каждой — список `Position` |
| `Metrics` / `PalletMetrics` | Агрегаты по паллетам и суммарно: **percolation** (доля объёма коробок в объёме под параллелепипедом по габаритам), **min_support_ratio**, **height_balance**, центр масс, площади опоры и т.д. |

Формат входных CSV в `tests/`: столбцы `SKU`, `Quantity`, `Length`, `Width`, `Height`, `Weight`, `Strength`, `Aisle`, `Caustic`.  
Формат ответа: заголовок `Pallet,SKU,x,y,z,X,Y,Z`.

## Мультипаллеты и `multitests`

- Склейка: `TestData operator+(a, b)` объединяет коробки и **суммирует** `pallet_count`; заголовки упаковки должны совпадать.
- Каталог `multitests/<id>/*.csv`: файлы с теми же именами, что в `tests/` (`3.csv`, `7.csv`, …); `load_multitest_combined` сортирует имена по **числовому** id и склеивает через `+`.
- Генератор: `bin/generate_multitests` — для каждого id из `tests/` создаёт подпапку в `multitests/` со случайным набором из **2–6** исходных CSV (см. `--seed`, `--tests`, `--out` в `src/generate_multitests.cpp`).
- Метрика **`height_balance`** ∈ [0, 1]: чем ровнее высоты штабелей на разных паллетах, тем выше; в скоре LNS/GA участвует как `score_height_balance_mult * height_balance`.

## Алгоритмы (идея)

- **`HeightHandler`**: карта занятых прямоугольников по XY и высоты «ландшафта»; быстрый перебор кандидатных углов и запрос максимума высоты в прямоугольнике.
- **`GreedySolver`**: порядок коробок (после разбиения по паллетам), для каждой — перебор ориентаций и позиций, выбор по минимальной итоговой высоте верха (tie-break по координатам).
- **`LNSSolver`**: случайные мутации кодировки (порядок, поворот, привязка к углу), пересчёт затронутых паллет симуляцией как у жадного размещения; **принимаются только не ухудшающие** шаги по скору из заголовка (отжиг в текущей реализации не используется).
- Остальные солверы и связь с литературой — в **[ALGORITHMS.md](ALGORITHMS.md)**.

## Результаты и публикации

Количественные сравнения алгоритмов, графики и обсуждение компромисса «заполнение / устойчивость / время» приведены в **`papers/`** (LaTeX, PDF собирается локально). В реферате (`papers/00-abstract.tex`) кратко зафиксировано, в частности: на **436** реальных задачах детерминированные эвристики дают порядка **65–66 %** коэффициента заполнения при времени **менее 10 мс**, метаэвристики — баланс **63–67 %** заполнения и **60–69 %** по минимальной опоре при более длительном счёте. Актуальные цифры после изменений кода лучше снимать повторным прогоном `./bin/main` и анализом `answers/metrics.csv` и сводной таблицы в консоли.
